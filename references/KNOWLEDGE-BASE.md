# Knowledge Base System

Local memory architecture replacing the monolithic 3-file system.
Each piece of knowledge lives in its own file, searchable by a dedicated sub-agent.

---

## 1. Directory Layout

```
kb/
├── raw/           # Unverified ideas, brainstorming, rough plans
├── archive/       # Rejected/falsified items with reason + evidence
├── curated/       # Validated items, confirmed decisions
└── _index.md      # Auto-maintained tag→file lookup (lightweight)
```

State transitions: new → `raw/` → tested false → `archive/` / tested valid → `curated/`.
Items move (file is moved between dirs), never deleted.

---

## 2. Note Format

Every note is a single Markdown file with YAML frontmatter.

```markdown
---
id: <YYYYMMDD-HHMMSS>-<slug>
title: "<short descriptive title>"
status: raw | archived | curated
domain: research | code | design | document | analysis | project | other
tags: [<tag1>, <tag2>]
created: <ISO-8601>
updated: <ISO-8601>
links: [<id-of-related-note>, ...]
evidence: [<E# references>]
confidence: <0.0–1.0>
archived_reason: "<why rejected>"       # only in archive/
decision_id: "<D# if linked>"           # optional
---

# <Title>

<Body: the actual content — idea, finding, decision, etc.>

## Evidence

<E# entries with fetched_at, method, and domain-specific fields>

## Links

- [[<related-note-id>]]: <relationship description>
```

### 2.1 Field Rules

| Field | Required | Notes |
|---|---|---|
| `id` | YES | Timestamp + slug ensures uniqueness. Example: `20260303-143000-connection-pooling` |
| `title` | YES | Human-readable, 5–80 chars |
| `status` | YES | Must match parent directory |
| `domain` | YES | From `task_type` in Start Contract |
| `tags` | YES | Minimum 1 tag. Use domain-specific tag names from DOMAIN-TABLES.md § 3 |
| `created` | YES | Set once at creation |
| `updated` | YES | Updated on every edit |
| `links` | NO | Bidirectional: if A links to B, B should link back to A |
| `evidence` | NO | Required for `curated/` notes; optional for `raw/` |
| `confidence` | YES | 0.0–1.0. Must change when evidence changes |
| `archived_reason` | CONDITIONAL | Required when status = archived |
| `decision_id` | NO | Links to D# in work-log for traceability |

### 2.2 Naming Convention

Filename = `<id>.md`. Example: `20260303-143000-connection-pooling.md`

No spaces, no special characters beyond hyphens. Lowercase only.

---

## 3. Tag Taxonomy

Tags serve as the primary discovery mechanism. Use a flat namespace with prefixes.

| Prefix | Meaning | Example |
|---|---|---|
| `topic/` | Subject matter | `topic/db-pooling`, `topic/battery-market` |
| `type/` | Content type | `type/hypothesis`, `type/benchmark`, `type/decision` |
| `loop/` | Feedback loop number | `loop/3`, `loop/7` |
| `gap/` | Linked to gap register | `gap/GAP-1` |
| `priority/` | Urgency | `priority/high`, `priority/low` |

Additional domain-specific tags are permitted and encouraged.

---

## 4. Cross-Referencing (Bidirectional Links)

Use `[[note-id]]` syntax inside both frontmatter `links` array and body text.

```markdown
This finding contradicts [[20260303-140000-polling-approach]], which assumed
polling was sufficient for real-time updates.
```

When creating a link A → B, also add A to B's `links` array.
If B is in a different status directory, the link still works (search agent
resolves paths).

---

## 5. Index File (`_index.md`)

Lightweight lookup table, auto-regenerated by the agent after each note
creation/move/deletion. NOT for reading into context directly — exists so the
search agent can do fast tag lookups without scanning all frontmatter.

```markdown
# KB Index
Updated: <ISO-8601>
Total notes: <N> (raw: <n1>, archive: <n2>, curated: <n3>)

## Tag Index
- topic/db-pooling: [20260303-143000-connection-pooling, 20260303-150000-pool-size-test]
- type/hypothesis: [20260303-143000-connection-pooling]
- loop/3: [20260303-150000-pool-size-test]
...
```

## 6. Search Sub-Agent Protocol

The main agent delegates knowledge lookups to a search sub-agent rather than
loading entire files into context.

### 6.1 When to Use the Search Agent

- Before creating a new proposal: "Is there prior work on this topic?"
- During DIAGNOSE step: "What are the weakest items?"
- During TEST step: "What evidence exists for/against this?"
- During META-ASSESS: "Am I retreading ground?"
- When resuming a session: "What was the latest state?"

### 6.2 Query Protocol

The main agent invokes `run_subagent` with a structured objective:

```
run_subagent(
  task_name="KB Search: <brief description>",
  subagent_type="general_purpose",
  objective="""
  Search the knowledge base at kb/ for information about: <query>

  SEARCH PROCEDURE:
  1. Read kb/_index.md for tag-based lookup
  2. For matching tags, read the relevant note files
  3. If tag search insufficient, grep across kb/ for content matches
  4. For link-following: read linked notes up to 2 hops

  RETURN FORMAT:
  - Matching notes: list of {id, title, status, confidence, summary (2-3 lines)}
  - Relevant evidence: E# entries from matching notes
  - Contradictions: any conflicts between matching notes
  - Gaps: related GAP-# entries if found

  CONSTRAINTS:
  - Read at most 10 notes per query
  - Return at most 5 most relevant results
  - If nothing found, say so explicitly — do not fabricate
  """
)
```

### 6.3 Response Integration

When the search sub-agent returns results:

1. If relevant prior work found → reference it (don't duplicate)
2. If contradiction found → log it, add to current loop's DIAGNOSE
3. If gap found → check if current loop can address it
4. If nothing found → proceed with new note creation

### 6.4 When NOT to Use the Search Agent

- KB has fewer than 10 notes: just read `_index.md` and relevant files directly
- You just created the note in the current loop iteration (it's in context)
- Quick status check: read `_index.md` header for counts

## 7. Note Lifecycle Operations

### 7.1 Create

```
1. Generate id from current timestamp + slug
2. Write file to kb/raw/<id>.md with full frontmatter
3. Add any [[links]] to related notes; update those notes' links arrays
4. Update kb/_index.md
```

### 7.2 Promote (raw → curated)

```
1. Move file from kb/raw/ to kb/curated/
2. Update status field in frontmatter to "curated"
3. Update `updated` timestamp
4. Ensure evidence section is complete
5. Update kb/_index.md
```

### 7.3 Reject (raw → archive)

```
1. Move file from kb/raw/ to kb/archive/
2. Update status to "archived"
3. Add `archived_reason` to frontmatter
4. Update `updated` timestamp
5. Update kb/_index.md
```

### 7.4 Revise

```
1. Update fields in-place (body, evidence, confidence, tags)
2. Update `updated` timestamp
3. If status changes, move file to correct directory
4. Update kb/_index.md if tags changed
```

---

## 8. Registers

Uncertainty Register and Data-Gap Register live as special notes in `kb/curated/`:

- `kb/curated/UNCERTAINTY-REGISTER.md` — one table, updated each loop
- `kb/curated/GAP-REGISTER.md` — one table, updated when gaps found

These are the only "living documents" that get edited in-place rather than
being append-only. Tag them `type/register` for easy lookup.

---

## 9. Session Continuity

When resuming a session:

```
1. Read kb/_index.md → understand current state (counts, latest tags)
2. Search sub-agent: "What were the last 5 notes created/updated?"
3. Read work-log.md incrementally (§ 2.3 of SKILL.md)
4. Continue feedback loop with full context
```

---

## 10. Migration from Monolithic Files

If starting from an existing 3-file setup:

```
1. Parse each item in working-raw.md → create individual kb/raw/<id>.md
2. Parse each item in proved-archive.md → create individual kb/archive/<id>.md
3. Parse each item in proved-curated.md → create individual kb/curated/<id>.md
4. Extract Uncertainty/Gap registers → kb/curated/UNCERTAINTY-REGISTER.md, GAP-REGISTER.md
5. Build kb/_index.md from all created notes
6. Delete the 3 monolithic files
```
