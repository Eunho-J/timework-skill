# Forced Feedback Loop

**[English](README.md)** | **[한국어](README.ko.md)** | **[日本語](README.ja.md)** | **[中文](README.zh.md)**

AIエージェントが与えられた予算（時間またはループ回数）内で一切停止せず、自己フィードバックループを回し続けて作業品質を引き上げるスキル。

## 解決する課題

- AIが一度ざっと見て「完了です」と止まってしまう
- 「次はこうしましょうか？」のような実行を伴わない空約束
- 作業履歴がコンテキストにしか残らず、セッション終了で消える
- 自分の結論を検証なしに確定する確証バイアス
- 既存情報の使い回しだけで探索範囲が狭まっていく
- アイデアが蓄積するほどファイルが肥大化し検索不能になる

## 基本原則

1. **予算が尽きるまで絶対に止まらない** — `min_required_minutes` または `min_required_loops` の間、能動的な作業のみ実行
2. **毎ループ新しい仮説を生成する** — 既存情報の組み合わせ（収束）＋ 未知の探索（発散）の両方向
3. **すべてをファイルに記録する** — `work-log.md` に逆時系列レポートを積み上げ、セッション間の連続性を保証
4. **個別ノート＋検索でナレッジを管理する** — Obsidian形式のKBにアイデアを個別ファイルで保存、検索サブエージェントが必要なものだけ取得

## スキルファイル構成

```
forced-feedback-loop/
├── SKILL.md                        # コア実行ポリシー（起動時にロード）
└── references/
    ├── DOMAIN-TABLES.md            # ドメイン別テーブル（遅延ロード）
    ├── PROHIBITIONS.md             # 27項目の禁止事項（遅延ロード）
    └── KNOWLEDGE-BASE.md           # KBシステム: ノート形式、タグ、検索プロトコル
```

## 実行時の生成ファイル

エージェントが実行中に作業ディレクトリへ生成するファイル:

```
<作業ディレクトリ>/
├── work-log.md                     # 作業履歴全体（逆時系列レポートスタック）
└── kb/                             # Knowledge Base（Obsidian形式）
    ├── _index.md                   # タグ→ファイル自動インデックス
    ├── raw/                        # 未検証のアイデア、下書き
    │   ├── 20260303-143000-xxx.md
    │   └── ...
    ├── archive/                    # 棄却された項目（理由付き）
    │   ├── 20260303-150000-yyy.md
    │   └── ...
    └── curated/                    # 検証済み項目
        ├── 20260303-160000-zzz.md
        ├── UNCERTAINTY-REGISTER.md # 不確実性レジスタ
        └── GAP-REGISTER.md         # データギャップレジスタ
```

### なぜ個別ノートか？

モノリシックな3ファイル方式はアイデアの蓄積に伴い肥大化します:

- 全体をコンテキストにロードするとトークンの無駄
- 関連項目のみの取得が不可能
- セッションが長くなるほど検索効率が悪化

KBシステムはアイデア・決定・エビデンスを個別Markdownファイルとして保存し、YAMLフロントマター（タグ、状態、信頼度等）でメタデータを付与。検索サブエージェントが必要なものだけを正確に取得できます。

### 検索サブエージェント

エージェントがKBから情報を取得する際、全体を読み込まず検索サブエージェントに委任します:

1. `_index.md` によるタグベース検索
2. マッチしたノートファイルの読み込み（最大10件）
3. 上位5件を {id, タイトル, 状態, 信頼度, 要約} で返却
4. 関連エビデンス、矛盾点、ギャップも併せて返却

KBのノートが10件未満の場合、サブエージェントなしで直接読み込みます。

## 使い方

ダウンロードしたzipを展開し、エージェントのスキルディレクトリに配置します。

| エージェント | パス |
|---|---|

### 実行例

**リサーチ:**
```
30分かけて「韓国のEVバッテリー市場動向」を調査してください。
```

**コード開発:**
```
20分かけてこのAPIサーバーのレスポンス速度を改善して。現在のp99レイテンシは500ms。
```

**デザイン:**
```
15分かけてこのダッシュボードレイアウトのユーザビリティを改善してください。
```

**ドキュメント:**
```
25分かけてこの技術提案書の論証構造を強化してください。
```

**回数ベース（時間制限なし）:**
```
フィードバックループ10回でこの関数のエラーハンドリングを改善して。
```

**両方（時間＋回数）:**
```
最低20分以上かつループ5回以上でこのクエリを最適化して。
```

時間、ループ回数、または両方を指定できます。終了条件が満たされるまでフィードバックループを回し続けます。

### 終了モード

| `min_required_minutes` | `min_required_loops` | 動作 |
|---|---|---|
| 設定 | — | **時間ベース**: 時間切れまで実行 |
| — | 設定 | **回数ベース**: 正確にN回ループ実行 |
| 設定 | 設定 | **両方**: 両条件が満たされるまで実行 |

どちらも未設定の場合、デフォルト5分が適用されます。

### 収束 × 発散 デュアル提案

各ループの提案ステップで2つのモードを交互に使用します:

| モード | 方向 | 説明 |
|---|---|---|
| **収束 (Convergent)** | 既知 → 新仮説 | KBの2つ以上の発見を組み合わせて新たな提案を生成 |
| **発散 (Divergent)** | 未知 → 新仮説 | KBがまだ答えられない隣接する問いを特定し、新情報を積極的に探索して検証 |

連続3ループ中、最低1回は発散提案を含む必要があります。収束のみだと探索範囲が狭まります。
発散提案は `task_goal` に対する関連性チェックを通過する必要があるため、テーマから逸脱しません。

### work-log.md の読み方

レポートは上から最新順に積まれます。各レポートの先頭行にそのレポートの行数が記載されており、必要な分だけ段階的に読めます:

```
=== Report #3 | lines: 6 | elapsed: 14:30 | type: feedback ===   ← 1行目
DIAGNOSE: Weakest decision is D2 (confidence 0.4).
PROPOSE: Inversion — what if we use push instead of pull?
TEST: grep codebase for event-driven patterns → 3 hits.
UPDATE: D2 confidence raised to 0.65.
META: Progressing. Next loop targets D4.                         ← 6行目
---
=== Report #2 | lines: 4 | elapsed: 10:00 | type: feedback ===
...
```

**`lines:` の基準:** ヘッダ行（含む）から最終コンテンツ行（含む）まで。`---` セパレータとレポート間の空行はカウントに含みません。
エージェントはレポート作成直後にこの値を検証し、スクリプティング環境ではlintチェックを実行します。

過去レポートの参照には相対座標（`K-reports-below`, `line N below`）を使用するため、ログの整理・圧縮後も参照が壊れません。

## マルチセッション

前回セッションの `work-log.md` と `kb/` ディレクトリが残っていれば、次のセッションで続行できます。エージェントが `kb/_index.md` を読み、検索サブエージェントで最近のノートを確認した上で、前回の記録を復元しフィードバックループを続けます。

```
昨日やっていたバッテリー市場リサーチをあと20分続けて。
```

## 互換性

[Agent Skills オープン仕様](https://agentskills.io/specification)に準拠し、この仕様をサポートする全てのエージェントで動作します。
